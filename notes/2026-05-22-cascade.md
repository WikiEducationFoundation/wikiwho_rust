# 2026-05-22 — matching cascade port

**Goal:** port the matching cascade (paragraphs → sentences → tokens) from
`wikiwho.py:207-691` and wire it into a `Article::analyse_revision`
entry point so the parity-check binary drives the real algorithm
(not just the tokenizer).

**Parity:**
- revisions: 15 / 16 (93.75%) — unchanged
- tokens: 876,775 / 973,970 (90.02%) — unchanged
- ms: 870 → 1026 (cascade allocation overhead; correctness-first, optimise later)

The headline number stays put because the cascade walks the same
splitter as `tokenize_revision`, and the parity-check still only
compares token *strings*. The cascade adds metadata (`token_id`,
`origin_rev_id`, `inbound`, `outbound`) that single-rev fixtures
fundamentally cannot validate. To move the metric we need
multi-revision fixtures (full rev_1..rev_N history per article) —
queued in `notes/decisions-needed.md`.

The cascade ALSO doesn't move the metric because the Myers diff path
(for `text_prev` non-empty) isn't implemented yet. The current code
hits `todo!()` if it gets there. Single-rev parity-check never gets
there.

**Issues encountered + resolutions:**

- *First length-shrink test text was too repetitive.* I built a
  "previous revision" with `"Lorem ipsum dolor sit amet. ".repeat(60)`
  expecting it to commit and then be length-shrunk on the next rev.
  It didn't commit — the token-density gate fired (avg word freq ~60,
  way above the 20.0 limit) and flagged the *first* rev as vandalism.
  That's correct algorithm behaviour; the test fixture was wrong.
  Switched to `(0..200).map(|i| format!("uniqueword{i} ")).collect()`
  which has 200 unique tokens (density 1.0) and 2600 chars (passes the
  >1000 PREVIOUS_LENGTH gate). Also added an explicit
  token-density-vandalism test using the original repetitive pattern.

- *Good-faith-move integration test would have hit the Myers `todo!()`.*
  Test text was `"x".repeat(50)` (one token), and the prev was a real
  cascade-committed first revision with ~200 sentences. The cascade
  would dutifully build `text_prev` from all 200 prev sentences and
  panic on the diff path. Worked around by hand-crafting a previous
  revision with `length: 1500` but no paragraph content (default
  Revision). The escape hatch is testable; the diff path stays
  unreachable until Myers lands.

- *Borrow checker around `revision_prev`.* The cascade needs to read
  `revision_prev.paragraphs.get(hash)` while mutating `article` (alloc
  paragraphs, push to `paragraphs_ht`, etc.). Solved by cloning
  `revision_prev` at the top of `analyse_revision` — the clone is
  a HashMap<Hash, Vec<u32>> + a Vec<Hash>, typically <10KB per
  revision. Acceptable for now; documented as a perf candidate when
  Obama-class processing comes online.

**Next session likely starts with:**

Myers diff on `&[u32]` interned token ids. That's the bottleneck for
real multi-revision processing — without it, any second revision
whose paragraphs don't fully reuse from prev panics via `todo!()`.
Plan:

1. **Myers in a new `diff` module.** Input: `&[u32]` for `text_prev`
   and `text_curr` (after interning the string tokens through a
   small `HashMap<&str, u32>` in `analyse_words_in_sentences`).
   Output: a transcript of `(Op, &str)` tuples — `Op::Keep`,
   `Op::Delete`, `Op::Insert`. This matches the Differ transcript
   `wikiwho.py:631-691` consumes.

2. **Wire the general case in `analyse_words_in_sentences`.** Replace
   the `todo!()` with a Myers call and the matching/deletion/insert
   loop from wikiwho.py:633-691. Note the "pop the first unmatched
   prev word with this value" semantics on duplicate tokens —
   spelled out as load-bearing for parity in ALGORITHM.md §4.3.

3. **Inbound / outbound recording.** After the cascade succeeds,
   walk `matched_paragraphs_prev` / `matched_sentences_prev` /
   `matched_words_prev` to bump `inbound` and `last_rev_id` on
   matched words, and walk `unmatched_sentences_prev` to append
   `outbound` on deleted words (`wikiwho.py:257-305`).

4. **Multi-revision integration tests.** Two-rev sequence with
   shared paragraphs, with diverging paragraphs (forces sentence
   level), with a mid-sentence edit (forces token level + diff).
   These won't make the parity number move, but they'll catch
   correctness regressions before the fixture upgrade.

5. **Fixture upgrade plan (separate from session 5).** Capture
   full revision history for the existing 16 fixtures so the parity
   binary can feed every rev to `analyse_revision` and compare the
   final state against the snapshot. This is a Python scripting
   chunk, not Rust — different shape of work, probably its own
   session.
