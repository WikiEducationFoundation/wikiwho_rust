- **`crates/wikiwho-server/src/whocolor_html.rs`** — new module with
  `inject_spans(html, tokens) -> InjectionResult`. Three-pass
  pipeline:
  1. Parse HTML with `html5ever` + `markup5ever_rcdom`.
  2. Flatten DOM into an event stream (`OpenTag`, `CloseTag`,
     `Text`, `Comment`, `Doctype`). Text events from inside
     skip-elements (`<script>`, `<style>`, `<head>`,
     `<noscript>`, `<template>`) carry an `inside_skip: true`
     flag.
  3. Concatenate non-skip text into a flat buffer; run a
     cursor-style matcher over it (case-insensitive, advances
     cursor on each match, tokens that never match are silently
     dropped — same shape as Python's `WikiMarkupParser.__set_token`).
  4. Re-emit events; in text events, splice in spans at the
     recorded match byte ranges.
  Plus `token_class_name(editor)` — md5(`0|<anon>`) for anons,
  user_id pass-through for registered. The class name uses
  `"editor-token token-editor-{X}"` (matches Dashboard's
  `ArticleScroll.js:28` selector). 12 unit tests covering case
  insensitivity, void elements (`<br>`), `<script>` content
  passthrough, cross-text-node matches, Unicode, and HTML escape
  round-trip.
  - `html5ever`, `markup5ever_rcdom`, `md-5` added to
    `wikiwho-server`'s deps.

**Design notes / issues encountered:**

- *Why three passes instead of in-place DOM mutation?* The original
  attempt walked the parsed tree and tried to inject spans by
  mutating text nodes' parent children vec. That ran into
  multiple-borrow issues against `Rc<RefCell<…>>` and made
  cross-text-node matching awkward (a token that doesn't match in
  the current text node should be checked against subsequent ones).
  The flat-text → matches → re-emit approach mirrors Python's
  approach more closely (where the entire wiki_text is a single
  buffer the cursor walks).

- *Case-folding can shift byte length.* First implementation
  computed `match_end = start + needle.len()`, which broke for
  CJK tokens where the cursor landed mid-character. The fix:
  `find_case_insensitive` now returns `(start, end)` in
  haystack bytes — it walks chars in the haystack, accumulating
  their lowercase forms into a prefix buffer, and stops at the
  exact char-aligned end of the matched region. Handles both
  shorter (ß→ss) and longer (İ→i̇) lowercase forms correctly.
  Covered by a Unicode test in `whocolor_html::tests`.

- *Why `extended_html` includes the full Parsoid `<!DOCTYPE
  html><html>…` shell?* PLAN.md §4.6's spec says the response is
  the HTML "Wikipedia would show" with spans injected. We pass
  the whole document through; the Dashboard's ArticleViewer
  scrapes its `<body>` contents via `__processHtml`
  (`ArticleViewerAPI.js:150`). Keeping the full shell makes the
  output usable as-is in a browser; if Dashboard wants just the
  body slice it does that on its end. (Same as production
  WikiWho's current output.)

- *Parsoid REST endpoint variation.* I picked `/api/rest_v1/page/
  html/{title}/{rev_id}` over the newer `/w/rest.php/v1/revision/
  {rev_id}/html` because PLAN.md §4.6 names the former
  explicitly. The two are aliases on WMF clusters but the alias
  may not exist on private wikis; we should revisit if/when we
  deploy to non-WMF deployments.

- *Editor name resolution graceful degradation.* If
  `resolve_users` fails (network blip, MW returning 5xx after
  retry budget), the handler logs a warning and proceeds with
  raw editor strings as both the name and class — so the
  response is still useful (it just shows user_ids instead of
  names in the present_editors / revisions chain). Mirrors
  Python's `WhoColor/utils.py:138-139` behavior on the `error`
  envelope path.

- *Cache-miss reuses the rev_content path.* The WhoColor handler
  doesn't have its own background ingest — it spawns the same
  `state.spawn_cache_miss` task the rev_content path uses. After
  it completes the article is on disk and any subsequent
  WhoColor request (or rev_content request, or anything) reads
  the persisted state. This means WhoColor benefits from any
  future incremental-ingest work for free.

- *`extended_html` is NOT byte-identical to production cache.*
  Different approach (we inject into Parsoid HTML directly;
  Python annotates wikitext then sends to MW's parser). Span
  placements should be at the same conceptual positions (per
  PLAN.md §4.6's accepted divergence), but the exact whitespace
  / attribute ordering / etc. of the surrounding HTML differs.
  A future session that wants byte-level WhoColor parity will
  need either to switch to Python's wikitext-injection approach
  (slower, requires Action API parse roundtrip) or to add a
  whocolor-parity fixture set captured at the same time as the
  rev_content fixtures.
